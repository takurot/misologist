## Overview

Misologist is an AI-assisted platform for home miso makers, small breweries, and fermentation researchers. It helps diagnose fermentation issues from photos, translate craft knowledge into fermentation chemistry, and manage miso batch records with Supabase.

## Current Features

### Emergency Fermentation Diagnosis

Upload a fermentation photo and Claude Opus 4.7 returns a structured diagnosis.

- **Mold identification:** Distinguishes likely white film yeast, blue mold, red yeast, and related cases.
- **Urgency levels:** `GREEN`, `YELLOW`, or `RED`.
- **Fermentation chemistry rationale:** Explains why the batch is or is not at risk.
- **Immediate actions:** Provides remediation and prevention steps.
- **Metadata-aware prompts:** Uses start date, temperature, storage location, soybean variety, koji ratio, and salt ratio when provided.

Photos are currently sent to the diagnosis API as base64 image data. The Supabase Storage bucket is defined in the schema, but the application does not yet persist uploaded photos to Storage.

### Batch Monitoring

Create batches, view batch detail pages, and request an AI-generated batch assessment.

- Batches are stored in Supabase.
- A batch assessment is generated on demand through the app API.
- The latest assessment is stored in `agent_sessions`.
- `next_action_at` can be populated from the model response.

This is not currently a long-running Claude Managed Agents integration or scheduled background worker. It is an on-demand API flow implemented in `app/api/agent-sessions/route.ts`.

### Craft Knowledge Translation

Enter workshop heuristics or traditional fermentation knowledge and receive a chemistry-grounded explanation from Claude Opus 4.7.

### Recipe Management and Reverse Design

Browse starter recipes, save custom recipes, and generate recipe suggestions from a target flavor profile.

- Starter templates are seeded through the Supabase schema.
- Saved recipes are stored in the `recipes` table.
- Recipe suggestions are generated by the app API and can be saved.

## Product Direction

This project started as a one-shot photo diagnosis tool and was redesigned around three core issues.

| Original issue | Why it was weak | Current direction |
|---|---|---|
| Single-photo diagnosis only | Vision-only analysis is not enough to justify Opus 4.7 | Diagnosis enriched with batch metadata and future longitudinal logs |
| A vague 0-100 fermentation score | Hard to defend the scoring logic | Explanations grounded in fermentation chemistry |
| Short synchronous agent tasks | Wastes the strengths of long-running async agents | Persistent batch records with on-demand monitoring, with room for scheduled agents later |

## Tech Stack

| Layer | Component | Responsibility |
|---|---|---|
| Frontend | Next.js 14 App Router | Diagnosis UI, knowledge translation, recipes, batch dashboard |
| AI Engine | Claude Opus 4.7 via Anthropic SDK | Vision diagnosis, fermentation reasoning, knowledge translation, recipe generation |
| API Runtime | Next.js Route Handlers | App API endpoints under `app/api/**` |
| Data | Supabase Postgres | Batches, recipes, agent session state, schema-defined logs |
| Storage | Supabase Storage | `miso-photos` bucket is defined; upload flow is not yet connected |
| Deployment | Vercel | Full-stack Next.js runtime |

## Setup

### Prerequisites

- Node.js 18+
- An Anthropic API key with Opus 4.7 access
- A Supabase project for batch and recipe features

### Environment Variables

Create `.env.local`.

For AI-only flows such as diagnosis, knowledge translation, and recipe generation:

```env
ANTHROPIC_API_KEY=your_anthropic_api_key
```

For batch and recipe persistence:

```env
NEXT_PUBLIC_SUPABASE_URL=your_supabase_url
NEXT_PUBLIC_SUPABASE_ANON_KEY=your_supabase_anon_key
```

`SUPABASE_SERVICE_ROLE_KEY` is not currently used by the application code.

### Initialize Supabase

For a hosted Supabase project, either run `docs/schema.sql` in the Supabase SQL Editor or use the migrations under `supabase/migrations/`.

The schema creates:

- `batches`
- `logs`
- `agent_sessions`
- `recipes`
- `miso-photos` Storage bucket

The current RLS policies are development-oriented `allow_all_*` policies and should be replaced before production use.

### Install and Run

```bash
npm install
npm run dev
```

The app starts at `http://localhost:3000`.

## Data Schema

```sql
batches:        id / name / started_at / recipe_json / status / created_at
logs:           id / batch_id / captured_at / photo_url / env_json / diagnosis_json / action_json / created_at
agent_sessions: id / batch_id / agent_state / last_action_at / next_action_at / created_at
recipes:        id / name / description / miso_type / koji_ratio / salt_ratio / soybean_variety / water_content / fermentation_duration / notes / is_template / created_at
```
