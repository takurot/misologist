# Misologist — 機能仕様書

**発酵診断・職人知識継承エンジン**
Anthropic Hackathon 2025 — Built with Opus 4.7
Version 1.0 | April 2025

> 「味噌職人の20年の経験を、Opus 4.7 が科学として語る」
> 発酵の失敗を防ぎ、職人の暗黙知を継承するAIエージェントプラットフォーム

---

## 1. 設計思想と差別化

### 1.1 修正した3つの問題

従来の「写真診断ツール」コンセプトから、以下3点を根本的に修正した。

| 修正前の問題 | 根拠 | 修正後の設計 |
|---|---|---|
| 単発写真診断 | Vision診断は小型モデルでも代替可能。Opus 4.7の必然性なし | 複数バッチ・複数年の長文ログを統合した縦断推論に変更 |
| 発酵スコア0-100 | 計算根拠が曖昧。審査員から「それどう計算するの？」と突っ込まれる | Opus 4.7が発酵化学的根拠付きで「なぜそのスコアか」を説明生成 |
| Managed Agentsを同期タスクに使用 | 写真診断は同期・短時間。非同期長期タスクというMCAの強みを無駄にする | バッチ監視エージェントとして非同期常時動作させる設計に変更 |

### 1.2 Opus 4.7の必然性（Why Not Smaller Models）

以下3点において、Opus 4.7は代替不可能：

- **長文コンテキスト統合：** 複数バッチ・複数月の発酵ログ全体を参照した比較推論
- **知識翻訳：** 「寒仕込みは旨味が出る」等の職人経験則 → メイラード反応・アミノ酸生成の科学的根拠への変換
- **不確実性の定量化と根拠生成：** 「このカビは除去すれば問題ない」という判断に至った発酵化学的プロセスを逐次説明する能力

## 3. コア機能仕様

### Feature 1 — 緊急発酵診断（Emergency Fermentation Triage）

審査員デモの核心。「今まさに失敗しかけている味噌を救う」シナリオを体験させる。

**入力**

- 発酵写真（PNG/JPEG、スマホ撮影直アップ対応）
- 環境メタデータ：仕込み日・現在温度・保管場所・使用した大豆品種・麹歩合・塩分率
- 任意：過去バッチのログ参照（長文コンテキスト活用）

**Opus 4.7の処理ロジック**

- Visionによるカビ種別判定：白カビ（産膜酵母/良性）vs 青カビ（ペニシリウム/危険）vs 赤カビ（ロドトルラ/要確認）
- 判定根拠の生成：色相・テクスチャ・発生パターン・発酵ステージからの複合推論
- 発酵化学的説明：なぜそのカビが問題/問題でないかのメカニズム解説
- 過去バッチとの比較：「3ヶ月前の同条件バッチと比較してカビ発生が2週間早い」等の縦断推論

**出力**

- 緊急度レベル：GREEN（問題なし）/ YELLOW（要経過観察）/ RED（即時介入）
- 具体的アクション指示：除去手順・消毒方法・再発防止策（ステップバイステップ）
- 発酵化学的根拠：判断に至った科学的プロセスの説明（Opus 4.7の差別化ポイント）

---

### Feature 2 — バッチ監視エージェント（Async Batch Watcher）

Claude Managed Agentsの本領発揮。仕込んだ日から完成まで、バックグラウンドで非同期常時監視。

**エージェント設計**

- バッチ1つ = 1エージェントセッション（最長12ヶ月の長期タスク）
- 毎日定刻に「今日のアクション確認」を生成（天地返し推奨・天候対応・塩嘗め時期）
- 環境センサー統合オプション：温湿度データをエージェントが自動取得・判断に組み込む

**ユーザー体験**

- 朝の通知：「今日は天地返しのタイミングです。理由：気温上昇により上層と下層の熟成ムラが予測されます」
- 受動的管理：ユーザーは確認してOKを押すだけ。判断はエージェントが行う
- 完成予測：現在の熟成速度から完成日を動的に再計算・更新

---

### Feature 3 — 職人知識翻訳エンジン（Craftsman Knowledge Translator）

「Build From What You Know」評価軸に直撃する独自機能。職人の経験則をOpus 4.7が科学に翻訳する。

**動作フロー**

1. ユーザーが「寒仕込みにすると旨味が強くなる気がする」と入力
2. Opus 4.7が発酵化学・酵素活性・温度と反応速度の関係を参照し科学的根拠を生成
3. 「低温（5-10℃）下ではプロテアーゼ活性が緩やかに進行し、グルタミン酸が豊富なアミノ酸プロファイルが形成されます」等の回答

**応用：逆引き設計**

- 「酸味を抑えて甘みを強くしたい」→ 麹歩合・仕込み温度・熟成期間の最適パラメータを逆算提案

## 4. システムアーキテクチャ

| レイヤー | コンポーネント | 役割 |
|---|---|---|
| フロントエンド | Next.js 14 App Router | 写真アップロード・診断結果表示・バッチダッシュボード |
| AIエンジン | Claude Opus 4.7 | Vision診断・発酵推論・知識翻訳・根拠生成 |
| エージェント基盤 | Claude Managed Agents | 非同期バッチ監視・長期セッション管理・定期アクション生成 |
| データ | Supabase (PostgreSQL) | バッチログ・診断履歴・写真メタデータ |
| ストレージ | Supabase Storage | 発酵写真のバイナリ保存 |
| デプロイ | Vercel | サーバーレス実行環境・Edge Functions |

### データスキーマ（主要テーブル）

```sql
batches:       batch_id / name / started_at / recipe_json / status
logs:          log_id / batch_id / captured_at / photo_url / env_json / diagnosis_json / action_json
agent_sessions: session_id / batch_id / agent_state / last_action_at / next_action_at
```

## 6. 3分デモシナリオ（審査員向け）

「今まさに失敗しかけている味噌を救う」という感情的インパクトを中心に構成。

| 時間 | シーン | 内容 | 審査軸 |
|---|---|---|---|
| 0:00–0:30 | 問題提起 | 「味噌仕込みは年1回。失敗したら翌年まで取り返せない」。熟練職人の知識格差の現実を30秒で提示 | Impact 30% |
| 0:30–1:30 | 緊急診断 | 青カビ写真をドロップ。Opus 4.7が種別判定・化学的根拠・緊急アクション手順を20秒で生成。「なぜ危険か」の説明が他のAIとの差別化ポイント | Opus 4.7 Use 20% |
| 1:30–2:20 | エージェント | バッチ監視ダッシュボードを表示。3ヶ月分のバッチ履歴と今日のアクション通知。エージェントが非同期で長期タスクを委譲された状態をライブ表示 | Demo 25% |
| 2:20–3:00 | 知識翻訳 | 「寒仕込みが旨い理由を教えて」→ アミノ酸・プロテアーゼ活性の科学的説明。職人経験則をAIが科学言語に変換するシーン | Depth 20% |

## 8. リスクと対策

| リスク | 発生確率 | 対策 |
|---|---|---|
| Managed Agentsの実装工数が想定超過 | 高 | Feature 2はデモ用モックで代替可能。Feature 1とFeature 3を先行完成させてから取り組む |
| Opus 4.7のカビ診断精度が低い | 中 | プロンプトに発酵化学の詳細コンテキストを埋め込む。プロンプトエンジニアリングで対応 |
| 5日間でUI品質が低くなる | 中 | shadcn/uiコンポーネントを最大活用。デモは機能を絞り「動くこと」を最優先 |
| Supabaseの設定・認証工数 | 低 | 初日に認証なしのPublic APIで実装開始。認証は最終日以降に追加 |

## 9. 審査基準への対応戦略（審査員の問いへの回答）

審査員が実際に問う4つの問いに対し、Misologistがどう答えるかを明示する。

### Impact (30%) — "Who benefits, and how much does it matter?"

味噌の国内市場規模は年間約800億円、家庭仕込み愛好家は推定50万人以上。職人技術の後継者不足は産業課題として認知されている。Misologistは「年1回しか仕込めない」という発酵の不可逆性ゆえに、失敗コストが極めて高い領域に介入する。1バッチの救済 = 半年〜1年分の努力と材料費の保全。

### Demo (25%) — "Is it genuinely cool to watch?"

青カビ写真を1枚ドロップして20秒以内に「なぜ危険か」の発酵化学的根拠が流れ出るシーンは、技術的驚きと感情的インパクトを同時に持つ。「AIが診断した」ではなく「AIが理由を語った」という体験の違いがデモの核心。

### Opus 4.7 Use (20%) — "Did they surface capabilities that surprised even us?"

職人の経験則を発酵化学に翻訳するという用途は、Anthropic社内でも想定外の領域である可能性が高い。単なるVision診断ではなく、**ドメイン暗黙知 × 科学的根拠生成 × 長文履歴統合**という3軸の組み合わせが「基本統合を超えた」使い方に該当する。

### Depth & Execution (20%) — "Did the team push past their first idea?"

仕様書セクション1.1に明示した通り、最初のアイデア（単発写真診断）を解体し、3つの根本問題を特定・修正した上で現設計に至っている。このピボットの過程をREADMEとデモ動画の冒頭で明示的に語ることで「wrestled with（格闘した）」ナラティブを審査員に伝える。
